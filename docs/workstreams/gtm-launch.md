# GTM and Launch Workstream

## Goal

Turn Operator OS from a download page with interesting claims into a clear public product story.

## Launch positioning

**Operator OS is a local-first operating system for AI agents.**

It should be positioned as:
- lightweight
- fast to start
- locally controlled
- extensible through tools, memory, MCP, channels, and automation

It should not be positioned yet as a fully mature hosted multi-tenant SaaS platform.

## Core blockers

### Critical
1. Strong public claims, especially footprint and startup speed, need current validation.
2. The current web surface is still mostly download-first rather than narrative-first.
3. Launch messaging must stay aligned with actual product maturity.

### Important
4. FAQ and objection-handling need to be written before opening beta.
5. Proof points must be selected carefully to avoid overclaiming.

## Public beta promise

A fast, local-first agent runtime you can run yourself, configure directly, and extend with tools and automations.

## Avoid promising

- fully managed hosted platform experience
- seamless managed services across every capability
- enterprise-grade multi-tenant control plane at launch

## Required deliverables

- launch narrative
- homepage outline
- proof-point validation list
- FAQ themes
- objections and rebuttals

## Two-week execution sequence

### Week 1
- validate core public claims
- freeze the launch narrative
- define homepage structure and core proof points

### Week 2
- rewrite current public page around narrative
- draft FAQ and beta explanation language
- align docs, releases, and repo descriptions with the same positioning

## Success metric

A new visitor can understand what Operator OS is, why it matters, and what stage it is in within 30 seconds.
